## Ce qui a été fait dans cette session

### Fix 1 — `is_solo_promote` universel (commit `eb43a619`)

**Problème** : le launcher proposait `consolidate_parallel_runs.py` pour le
run `CONSTITUTION_RUN_2026_04_11_LEARNER_STATE_R01`, pourtant solo et déjà
promu via `patch_lifecycle` (STAGE_07 atteint). Le script multi-runs était
erroné dans ce contexte.

**Correction appliquée dans `tmp/pipeline_launcher.py`** :
- `is_solo_promote` se déclenche pour **tout run dont `len(eligible_runs) == 1`**,
  quelle que soit sa `current_stage`.
- `_build_consolidation_stage_prompt` adapte le texte selon le stage réel :
  - `STAGE_07` → drive direct vers **STAGE_08_PROMOTE_CURRENT**.
  - `STAGE_06` / `STAGE_06B` → drive vers **STAGE_07 puis STAGE_08**.
- `consolidation_dry_run_cmd` et `consolidation_cmd` sont vides en mode solo.

### Fix 2 — Slots 2-N débloqués pour `open_new_run` (commit `eb43a619`)

**Problème** : en mode `--parallel-runs 4`, les slots 2-4 étaient vides car
le guard `open_new_run` excluait tous les scopes constitution dès qu'une
consolidation était présente (même déjà affectée au slot 1).

**Correction appliquée** :
- `pipelines_with_consolidation_slot: set[str]` suit les pipelines dont le
  slot consolidation a déjà été assigné.
- Le guard ne bloque plus si `pid in pipelines_with_consolidation_slot`.
- Les slots restants proposent `open_new_run` avec :
  - `parallel_with_consolidation: true`
  - `isolation_note` explicitant que les outputs doivent rester sous
    `runs/<new_run_id>/` jusqu'à fin de consolidation.
